# First-Run Onboarding Design

## Goal

Provide an interactive setup experience when wuu is used for the first time,
so users can configure their provider (type, base URL, API key, model) and
color theme without manually editing JSON files.

## Trigger

| Scenario | Behavior |
|----------|----------|
| `wuu` (no config found) | Auto-enter onboarding inside TUI |
| `wuu init` | Force re-run onboarding (overwrite existing config) |
| `wuu init --force` | Same as above, kept for backward compat |
| Config exists | Skip onboarding, launch normal TUI |

Detection: `config.Load()` returns "config not found" error → enter onboarding.

## Onboarding Steps

### Step 1 — Provider Configuration

1a. **Provider type** (list select):
- OpenAI
- Anthropic
- OpenAI-Compatible (third-party / self-hosted)

1b. **Base URL**:
- OpenAI → pre-filled `https://api.openai.com/v1`
- Anthropic → pre-filled `https://api.anthropic.com`
- OpenAI-Compatible → blank text input

1c. **API key** (password input):
- Masked display (`sk-...xxxx`)
- Stored globally in `~/.config/wuu/auth.json` (mode 0600)

1d. **Model** (list select + custom option):
- OpenAI: gpt-4.1, gpt-4.1-mini, gpt-4.1-nano, [custom]
- Anthropic: claude-sonnet-4-20250514, claude-3-5-haiku-latest, [custom]
- OpenAI-Compatible: [custom text input only]

### Step 2 — Theme Selection

- Dark (default)
- Light (placeholder — renders same as dark for now)

## Storage

### `~/.config/wuu/auth.json` (global, mode 0600)

```json
{
  "keys": {
    "openai": "sk-proj-...",
    "anthropic": "sk-ant-..."
  }
}
```

Keyed by provider name as it appears in `.wuu.json`.

### `.wuu.json` (project-level, generated by onboarding)

Standard config as today, but `api_key` field is omitted. The provider
factory resolves keys in order: `api_key` field → `api_key_env` env var →
`auth.json` global store.

### `~/.config/wuu/config.json` (global preferences)

```json
{
  "theme": "dark",
  "has_completed_onboarding": true
}
```

## Architecture

### New Files

| File | Purpose |
|------|---------|
| `internal/tui/onboarding.go` | Bubble Tea model for the onboarding flow |
| `internal/config/auth.go` | Global auth store (read/write `auth.json`) |
| `internal/config/global.go` | Global preferences (read/write `config.json`) |

### Modified Files

| File | Change |
|------|--------|
| `cmd/wuu/main.go` | `runTUI`: detect missing config → run onboarding TUI first, then reload and launch normal TUI. `runInit`: delegate to onboarding TUI instead of writing template. |
| `internal/providerfactory/factory.go` | `resolveAPIKey`: add fallback to `auth.json` when `api_key` and env var are both empty |
| `internal/tui/app.go` | No change needed — onboarding is a separate `tea.Program` run |

### Onboarding Model

```go
type onboardingStep int

const (
    stepProviderType onboardingStep = iota
    stepBaseURL
    stepAPIKey
    stepModel
    stepTheme
    stepDone
)

type OnboardingModel struct {
    step         onboardingStep
    cursor       int
    providerType string   // "openai" | "anthropic" | "openai-compatible"
    baseURL      string
    apiKey       string
    model        string
    theme        string
    textInput    textarea.Model
    width, height int
    err          error
}
```

### Data Flow

```
main.go: runTUI()
  → config.Load() fails (no config)
  → run OnboardingModel as tea.Program
  → user completes steps
  → onboarding writes:
      ~/.config/wuu/auth.json
      .wuu.json
      ~/.config/wuu/config.json
  → tea.Program exits with OnboardingResult
  → main.go reloads config.Load()
  → builds StreamRunner
  → runs normal TUI Model
```

### Key Resolution Order (updated)

```
resolveAPIKey(provider):
  1. provider.APIKey (explicit in config)
  2. os.Getenv(provider.APIKeyEnv)
  3. auth.LoadKey(home, providerName)  ← NEW
  4. error: no key found
```

## UI Interaction

- Arrow keys (↑↓) to navigate list items
- Enter to confirm selection
- Esc to go back one step
- Text input for base URL, API key, custom model
- API key input is masked (shows last 4 chars only)
- Each step shows a centered card with title and options
- Brand-colored header consistent with main TUI theme

## Error Handling

- Invalid API key format: warn but allow (we can't validate without a request)
- Empty base URL for third-party: block, show error message
- File write failure: show error, suggest manual config creation
- Esc on first step: exit the program
